# 系统架构与代码流程

## 一、系统位置

QVF 的标准输入是 `query + retrieved memories`。初始检索由外部 BM25、dense、hybrid 或其他 memory system 完成；QVF 不改变检索器内部索引。完成判断后，QVF 把 sidecar 和 controller action 交给外部 answer model。

## 二、一个请求怎样运行

1. `raw_input.py` 接收原始 query 和 retrieved memories，拒绝 benchmark、gold、judge 和 case metadata。
2. 可插拔 extractor 或 structured fast path 产生 entity、slot、value、时间、条件、基数与关系候选。
3. QVF 验证每个结构化字段能够回指原始 memory 的精确 source span；缺失时间和置信度保持 unknown，不补默认值。
4. `temporal_validity.py` 区分 observation/source/event/effective time、future plan、revocation 和未知顺序。
5. `semantic_relations.py` 验证 equivalent、additive coexistence、replacement、revocation、condition-bound、unrelated 或 unresolved 关系。
6. `memory.py` 在同实体、同 slot、同条件范围内建立 read-time 状态，scalar/set cardinality 不确定时不允许静默覆盖。
7. `answerability.py` 判断 query 所要求的 response dimension 是否有足够、相容、可作为当前状态使用的证据。
8. `query_risk_router.py` 决定 direct-preserve-first 或 QVF validity 路由；它不回答问题。
9. `decisions.py` 生成 memory roles、blocked/allowed IDs、model-facing sidecar 以及 answer/correct/retrieve/clarify 动作。
10. `controller.py` 汇总流程，并可在外部 retriever callback 存在时执行有上限的 retrieval repair。

## 三、核心文件

| 文件 | 作用 |
|---|---|
| `_pipeline_core.py` | 核心 schema、规范化和基础流水线函数 |
| `admission.py` | 将结构化 events/queries 规范化并验证输入 contract |
| `answerability.py` | response dimension、evidence sufficiency 和 attestation |
| `controller.py` | 公共 facade；组合 selective route、sidecar 和有界 retrieval repair |
| `decisions.py` | memory roles、model-facing payload、next action 与泄漏清理 |
| `lifecycle.py` | 组织 admission、memory store、packet 和 decision 生命周期 |
| `memory.py` | memory record、current/history/stale/condition role 和状态竞争 |
| `pipeline.py` | 应用集成首选入口 |
| `query_risk_router.py` | 可审计的确定性风险 Router 基线 |
| `raw_input.py` | 原始检索输入、精确 source-span provenance 和 fail-closed contract |
| `retrieval.py` | 从 store 构建 validity packets；不实现外部检索算法 |
| `semantic_relations.py` | 语义关系、target IDs 和 cardinality contract |
| `service.py` | 构建/运行 service request 并返回摘要和 sidecars |
| `temporal_validity.py` | 严格时间语义和 future/revocation/unknown 处理 |

## 四、Router 的真实状态

当前 active Router 是确定性、可审计的 baseline。它使用通用语言现象，例如 current/change/temporal/conditional/conflict/unknown/multi-session cues，并结合 request 中的结构化 evidence risk。它不包含职业、关系、地点等数据集领域专用词表，也不允许在线积累 benchmark 关键词。

Router 输出中的 confidence 明确标记为 `heuristic_priority_score_not_calibrated_probability`。它的作用是控制是否启动 validity-aware 路径，而不是直接判定答案。自适应语义 Router 仍需独立自然数据双标注、校准和 false-safe promotion gate。

## 五、为什么不是“只选最新”

较新的 observation 只有在同实体、同 slot、相同条件范围、scalar cardinality 且 replacement/effective-time 证据成立时，才可能替代旧值。否则系统保留共存、历史或 unresolved：

- future plan 不能直接作为 current；
- set-valued slot 允许多个值共存；
- condition-bound evidence 不能无条件扩展；
- 缺少时间或来源时 fail closed；
- retrieve/clarify 是上限 veto，低优先级 packet 不应重新授权 blocker rows。
