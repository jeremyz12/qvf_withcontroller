# 本周研究进展

## 1. 根据反馈完成的改进

### 从 benchmark adapter 转向通用原始输入 contract

新增 `raw_input.py`，使 QVF 的公开入口直接接收 query 和外部 retrieved memories。结构化抽取变成可插拔边界，并增加 exact source-span、provenance、missing-field 和 forbidden metadata 验证。benchmark adapter 因此不再是生产运行闭包的一部分。

### 修复时间语义

把 observed/source/event/effective time、future plan、revocation 和 unknown ordering 分开。系统不再采用无条件 recency；同 slot 更新必须有 cardinality 与 replacement/effective evidence 才能覆盖旧值。

### 扩展语义关系与基数

新增 equivalent、coexistence、replacement、revocation、condition-bound、unrelated 和 unresolved contract，避免同义改写、多值列表或条件值被误判为新旧版本。

### 加强动作一致性

answer、correct、retrieve 和 clarify 变成显式 controller actions；证据不足时 retrieve/clarify 作为安全上限，低优先级 packet 不应绕过 blocker。

### 检索器可组合测试

在冻结 BM25、dense 和 hybrid 输出上进行 paired controls。GPT-4o-mini 三条线上分别出现 `+4pp`、`+3pp`、`+3pp`，说明接口可以叠加不同上游检索结果。但两个非 GPT reader 的方向不一致，暴露出 reader-specific packet/action sensitivity。

## 2. 负结果和方向修正

- Claude Haiku 4.5：frozen BM25 `73 -> 69`，2 wins / 6 losses，未支持跨模型提升。
- Gemini 3.1 Flash-Lite：`61 -> 61`，5 wins / 5 losses，净效应为 0。
- Dual semantic provider：`42/48 = 87.5%`，虽 accepted cases 为 `42/42`，但低于预注册 overall 90% gate，因此 NO-GO。
- Mem0 tiny control：`3/6 -> 2/6`；Hindsight 1-case control：`0/1 -> 0/1`。样本太小，不能形成组合增益结论。

这些结果把下一步从“继续堆规则和扩大跑分”改成“先盲审 losses、建立 response-dimension supervision，再进行跨模型晋升”。

## 3. 当前研究定位

最稳妥的主贡献线是：

> QVF 是一个可插入不同检索器之后的、query-conditioned longitudinal-memory validity controller。它把 relevance 与 current validity 分离，并通过 provenance、时间、关系、条件和证据充分性约束，降低 stale/conflicting memory 被当作当前事实使用的风险。

MemConflict100 适合作为主要端到端贡献线；LongMemEval-S500 是 ordinary-memory non-degradation gate；STALE400 更适合作为 oracle-pair 机制上限；LoCoMo79 保持 supporting pilot。
