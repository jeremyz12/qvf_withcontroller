# 代码结构说明

本包保留当前 QVF controller 版本最相关的代码文件，并补入少量必要兼容封装，使随包附带的核心测试可以独立运行。完整仓库中还有大量实验脚本、报告生成脚本和历史 ablation 文件，这里没有全部复制。

## 关键文件

### `01_核心代码/src/qvf_validity_admission/public_answer_eval.py`

这是当前结果中最关键的文件，包含 public benchmark answer-eval 路径中的 controller 逻辑。它的作用不是单纯写 prompt，而是把 QVF 的检索后控制流程组织成多个结构化状态：

- 构建 direct baseline 的 memory context。
- 构建 QVF packed context。
- 决定 QVF 是否复用 direct target prompt，还是进入独立 QVF answer generation。
- 构造 source-history focus、condition-scope、temporal-resolution、habit-frequency 等 packet。
- 在 post-answer-audit controller 版本中尽量保持普通长期记忆 direct-equivalent。
- 执行 preflight、payload audit、target API 调用、judge API 调用、pairwise result 汇总。

本轮新增或强化的主要逻辑也在这里：

- `source_history_focus_context`：保留 source-history 中的通用时间或因果 focus。这里的规则不是 case phrase trigger，而是识别“相对时间窗口”和“因果/动机表达”等语言类别，再结合 source timestamp、query overlap 和 selected-history gate 决定是否进入 QVF packet。
- `source_history_answer_anchor_context`：保留 source span 中的答案锚点。
- `_selected_history_has_observed_boundary`：source-history route 的 timestamp promotion gate。
- `_source_history_requires_independent_target`：判断 source-history 场景是否需要让 QVF 独立生成。
- `_post_answer_audit_uses_direct_target`：判断 QVF 是否应复用 direct target prompt。

### `01_核心代码/src/qvf_validity_admission/query_risk_router.py`

这是更早一层的 query risk router。它根据 query 的 wording 和风险信号，把问题粗分到 current、conditional、transition、conflict 等路线。当前最终 controller 没有把它做成 learned router，而是使用通用规则和优先级 policy。

它的意义是把 query 层面的风险显式化，但最终是否进入 QVF，还需要结合 memory evidence、controller decision、retrieval feedback 和 packet availability。

### `01_核心代码/src/qvf_validity_admission/service.py`

这是 QVF service request 的入口封装。answer-eval 会通过它调用 QVF pipeline，得到 query report、read-time decision、compact validity packet 等结构。

### `01_核心代码/src/qvf_validity_admission/_pipeline_core.py`

这是底层 pipeline 组件之一，负责 QVF 的核心处理流程。它支撑 validity packet、read-time decision 和证据分类。

### `01_核心代码/src/qvf_validity_admission/decisions.py`

包含 QVF decision 相关的数据结构和判断逻辑。这里的 decision 不是模型输出的自然语言，而是 controller 后续使用的结构化状态。

### `01_核心代码/src/qvf_validity_admission/memory.py`

定义 memory record、validity 状态、时间字段、source 信息等基础结构。QVF 能区分 current/archive/stale/uncertain，很大程度依赖这里的数据组织。

### `01_核心代码/src/qvf_validity_admission/public_dataset_adapters.py`

用于把 LongMemEval、LoCoMo 等 public dataset 转换成统一 answer-eval item。它属于 benchmark adapter 层，不应把 adapter 修复直接当成 QVF core gain。

### `01_核心代码/src/qvf_validity_admission/public_extraction.py`

用于从 public dataset 的 conversation/history 中抽取候选 memory，生成 QVF service request。LoCoMo 的 coverage 问题很大一部分发生在这一层。

## 兼容封装文件

以下文件主要用于保持精简包的导入链和测试可运行，不是新的方法贡献：

- `admission.py`、`retrieval.py`、`lifecycle.py`、`pipeline.py`：把底层 `_pipeline_core.py`、`memory.py`、`decisions.py` 组织成可调用的 QVF pipeline。
- `answer_model_eval.py`、`analysis_pipeline.py`、`targeted_benchmark.py`、`integration_eval.py`：支撑 answer-eval、分析和集成测试路径。
- `answer_matching.py`：用于公开数据集 adapter 中的答案字段路径扫描。
- `stale400_adapter.py`：把 STALE400 case 转成 validity-admission request。

## 测试文件

### `01_核心代码/tests/test_public_answer_eval.py`

这是当前 controller 最重要的测试文件。新增测试覆盖了：

- source-history relative temporal phrase 是否保留。
- ER 这类短 domain term 是否不会被过滤掉。
- 通用 causal cue 是否能形成 focus row，避免把具体事件短语写成规则。
- answer-anchor context 是否进入 target prompt。
- 无 source timestamp 时是否不会提升为独立 source-history route。

## 当前 controller 的运行流程

简化流程如下：

1. 初始检索或 adapter 提供候选 memories。
2. QVF service 对 memories 做 validity annotation 和 evidence packet 构建。
3. Controller 判断当前 query 是否存在 validity/freshness/conflict/condition/temporal/source-history 风险。
4. 如果是普通长期记忆回忆，尽量复用 direct target prompt。
5. 如果风险成立，则构造对应 packet：
   - stale/current validity packet
   - timeline/conflict packet
   - condition-scope packet
   - scoped/temporal packet
   - source-history focus/anchor packet
6. Answer model 使用 packet 和 raw-memory fallback 生成答案。
7. Judge 或 post-answer audit 用于评估答案是否符合 evidence decision。

## 设计重点

当前系统最值得展示的点是：QVF 不是“多写几句 prompt”，而是把检索结果转成一组可审计的控制状态，然后决定答案生成是否需要受这些状态约束。

其中最关键的工程边界是：QVF 不替代检索，不写入新 memory，不使用 gold answer，不把 benchmark adapter 的修复当成 QVF core gain。
