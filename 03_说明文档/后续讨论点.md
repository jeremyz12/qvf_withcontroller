# 后续讨论点

## 1. Router 仍然是规则式 controller，而不是 learned policy

当前 router/controller 使用的是通用规则、cue 和 priority policy。它不是简单关键词匹配，但也不是 learned policy。当前设计的优点是可解释、可审计、便于控制 leakage；缺点是规则边界需要通过实验不断收窄。

目前 learned-router 的证据不足。已有 LongMemEval500 learned-router no-API ablation 显示 held-out net gain 为 0，因此暂时不建议把最终系统切换为 learned router。

可以讨论的问题：

- learned policy 是否应该作为 ablation，而不是主系统？
- 如果要训练 learned policy，训练标签应来自哪些真实 controller decisions，而不是 benchmark gold answer？
- learned policy 是否会牺牲 QVF 当前最重要的 claim boundary 和可解释性？

## 2. Source-history route 的收益和风险

本轮新增的 source-history focus/answer-anchor 对 LoCoMo 有帮助：

- LoCoMo10：Direct 0/10，QVF 3/10。
- LoCoMo79：Direct 41/79，QVF 46/79。

但初版 source-history route 太宽时，LongMemEval500 退化到 QVF 194/500，低于 Direct 196/500。加入 timestamp boundary promotion gate 后恢复为 QVF 203/500，高于 Direct 200/500。

可以讨论的问题：

- source-history route 是否应继续要求 timestamp/observed_at？
- 对没有 timestamp 的 source-history，是否只做 annotation，不做独立 QVF answer？
- LoCoMo coverage 是否应该通过 adapter/extraction 改进，而不是继续扩大 controller 路由？

## 3. 当前唯一明确 residual：答案过度列举

LoCoMo79 中唯一 QVF loss 是一个 over-inclusion 问题：QVF 答案包含正确项，但额外列出了多个活动，judge 按不够精确判错。

这个问题不是 router 是否进入 QVF 的问题，而是 answer policy 的问题。更通用的修复方向是：

- 当问题问单一事实或 gold-like expected answer 是单项时，答案应给最小充分回答。
- 当 memory context 中存在多个相关活动时，不应把所有相关项都列出来。
- 可以设计 `minimal_sufficient_answer_policy`，但需要先做 canary，避免影响需要多项列表的问题。

## 4. QVF 当前最强的贡献点

从结果看，最强贡献不是“所有长期记忆都更好”，而是：

- stale/current conflict 中避免 archive/stale memory 被当作当前事实。
- MemConflict 中处理条件范围、timeline/current pair 和 conflict evidence。
- source-history 场景中保留时间短语、因果触发语和 answer anchor。
- 对普通长期记忆保持 direct-equivalent，降低非必要干预。

## 5. 下一步优先级

建议下一步按这个顺序推进：

1. 做 LoCoMo79 唯一 loss 的 minimal-sufficient-answer canary。
2. 检查 MemConflict100 和 STALE400 在 timestamp gate 后是否不受影响；理论上不应受影响，但可以做 no-API payload/preflight check。
3. 如果 LoCoMo minimal-answer canary 通过，再跑 LoCoMo79 小复核。
4. 暂不推进 learned router 为主线，只作为 ablation/control。
5. 后续论文写法应强调 QVF 是 post-retrieval validity controller，而不是完整 memory system。
